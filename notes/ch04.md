# Chapter 4 - Lazy Evaluation

This chapter explores lazy evaluation and streams.

## OCaml translation

- I chose to use OCaml's built in [Lazy.t](https://ocaml.org/manual/5.4/api/Lazy.html) instead of reimplementing it. this felt prudent because the language has built in `lazy` syntax, which the book also described. I've done enough programming in Clojure and other languages that are lazy by default to understand how the deferred computations are constructed. I'd rather spend time understanding how lazy evaluation is applied to enable persistence and amortized costs in later chapters
- I added `of_list` and `to_list` to the STREAM signature for testing

## Observations

- The strict by default with opt-in laziness is a nice language feature in OCaml. I could see this being used to efficiently process large datastreams with succinct code that is exhaustively checked for type correctness. that's a nice feature for systems where incorrect code could crash a long-running expensive process
- laziness + memoization is a pattern that appears to be exploited in ch4 and beyond for amortized analysis. I can see through the match patterns of the text's SML and my OCaml translation that this enables general proof by cases for complexity analysis; the computational cost is incurred once and can be assigned at the point that forces it
- `reverse` forces the whole stream which makes it $O(n)$ but this eager approach is also noted in the book

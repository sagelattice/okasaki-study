### Splay heaps

- Splay heaps draw some inspiration from quicksort's partition concept. some access operations have an implicit restructuring (instead of explicit balance info) to maintain $O(\log\ n)$ amortized
- An emergent property of their behavior is that they structurally adapt their efficiency to the access patterns on them
- I found it interesting that splay heaps are not heap-ordered, and I had a wrong assumption that heaps required heap ordering generally.
- Finding the minimum of a splay heap requires recursing down to the leftmost leaf because they are BST ordered, which is $O(n)$ depending on the tree height
- `partition` is another case where a functional environment with a GC pays off in simplicity. pointer updates and memory management would be implementation overhead in the branching cases.
- I suspect in Rust I can have the language runtime manage some of this kind of overhead with [std::rc::Rc](https://doc.rust-lang.org/std/rc/struct.Rc.html). I want to avoid `unsafe` or lifetime manipulation on the first pass if possible. this would likely require returning a whole new value for the tree along with the query value as mentioned in the book

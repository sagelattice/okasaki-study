## OCaml translation

- ocaml uses dune as a build system; I set it up to enable importing the ORDERED sig instead of copying it
- I translated the tuples into curried params again; felt right at the time
- `let name = function` is syntax for single (implicit) arg pattern matching
- `match arg1, arg2, ... with` is syntax for pattern matching. [let rec](https://ocaml.org/manual/5.0/expr.html#sss:expr-localdef) puts the name into scope during definition so it can be self-referencing. it can also be used to define mutually recursive functions with `and`. they can be used together to define a recursive function with pattern matching dispatch
- [=](https://ocaml.org/manual/5.0/expr.html#ss:expr-operators) is structural equality, so that's right for immutability tests
- the book's HEAP definition has a comment about the exception; I defined it inline in the LeftistHeap functor for consistency with the book
- [or-patterns](https://ocaml.org/manual/5.4/patterns.html#sss:pat-or) let you collapse multiple matches to a logical or; Okasaki mentioned they are available in SMLNJ and OCaml has them too. I chose to do this minor refactor for clarity and to explore the language some

## Observations

### Leftist heaps

- rank + makeT enforce the $O(\log n)$ invariant for LeftistHeap (left rank $\geq$ right rank) which bounds the tree height. in the book they are defined separately; I added them in the functor definition which is hidden behind the HEAP type narrowing
- leftist heap ranks and [shor's algorithm](https://en.wikipedia.org/wiki/Shor%27s_algorithm#Classical_reduction) multiplicative groups exploit the algebraic structure $base^{period} - 1 = 0$. leftist heaps use $2^r$ to bound the growth of height (structure) and shor's algorithm uses $a^r \equiv 1 \pmod{N}$ to expose factors via GCD where a is a random coprime to N. the analogy is loose (structure vs computation) but this is the kind of cross-domain pattern recognition that builds intuition for algorithm design and complexity analysis

### Binomial heaps

- binomial heaps and the FFT both exploit binary decomposition on the structure of $2^k$. both of them collapse a naive $O(n)$ or $O(n^2)$ complexity into a log factor per element, but the heap uses it for structure where the FFT uses it for computation. the FFT does this through divide and conquer on DFTs. I find this shared algebraic structure fascinating and it brings me joy
- link enforces heap order and structure by making the heap with a larger root the left most child of the heap with a smaller root. this maps onto the visualization of ranks in the book. noting that the link order would be reversed for a max heap
- the recursive case of merge restructures the tree and generates the structure. the else (3rd) branch links the trees of equal rank which is the binary carry operation mentioned in the book
- the exercises for this chapter are interesting; 3.7 to optimize findMin from $O(\log\ n)$ to $O(1)$ looks fun, but I plan to complete the tour in ocaml and rust before doing these optimizations. optimizing the ocaml early may make it more difficult to look back and understand for rust. the same thinking applies to not storing explicit ranks for 3.6

### Red Black Tree (Sets)

- the presentation of `balance` is the cleanest I've ever seen and the or-pattern suggestion clarifies the idea even further. I don't think I had realized the cases collapsed to the same construction pattern (from different structural locations in the subtree) when I implemented RBT in my undergrad in C++.
- I now fully see what Okasaki meant when he was talking about how the GC does a lot of work in a functional environment. in OCaml the tree is rebuilt instead of mutated and all of the tracking  work is left to the GC. the pointer manipulation and in-place overhead in C++ was the primary source of struggle for me back then and I had to have debug flagged assertions on structure to get it working.
- search on a red black structure is always a joy to see; the elegance that falls out of the logical complexity of balance really shows why it's had staying power when teaching data structures and algorithms. the simplicity of the search and insertion logic lends itself to complexity analysis
- I annotated the subtree cases with comments; it was trivial to see them through the type constructor syntax in OCaml, especially since they were all defined in the same place. when I did this in C++ in my undergrad there was a lot more pointer chasing boiler plate that made it more challenging to see
- I had claude review the balance function's signature; it suggested that curried params with a match expr would eliminate the tuple allocation at call sites, and the tuple allocation for the match expr would be optimized by most compilers. I believe that would be worthwhile in a production setting but for a study I value the clarity and staying closer to the book's presentation more

## Persistence mechanism

- LeftistHeap: `type heap` is immutable as a recursive datatype through language guarantees
- BinomialHeap: `type heap` is also immutable through language guarantees on lists and tuples
- RBT: same language guarantees

There's a pattern here; the OCaml language affords persistence guarantees because the built in values used in the book are immutable. The datastructures are persistent by language definition, so I'll drop this section going forward unless there's a more nuanced persistence mechanism (perhaps lazy and amortization introduced by ch4 and ch5)
